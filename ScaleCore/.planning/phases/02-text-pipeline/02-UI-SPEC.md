# Phase 2 -- UI Design Contract

> Visual and interaction contract for Phase 2: Text Pipeline. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Classification: Backend-Only

Phase 2 introduces **zero new UI screens, components, or user-facing interactions**. All 12 requirements (PIPE-01 through PIPE-06, MEM-01 through MEM-03, BILL-01 through BILL-03) are server-side:

- Webhook message routing and tag-based filtering
- Redis message buffering (10s window)
- OpenRouter LLM API calls
- WhatsApp message sending with typing indicators and natural delay
- Conversation memory persistence in Supabase
- Wallet credit deduction via existing RPC

The "UI" for this phase is the WhatsApp conversation itself, rendered by the WhatsApp client -- not by ScaleCore.

**Consequence:** The design contract below documents the existing project design system for reference. No new visual work is required. If any incidental UI touches arise (e.g., agent status badge, wallet warning), they must follow these existing tokens.

## Copywriting Contract

No new user-facing copy is introduced in Phase 2. All interactions happen server-side via WhatsApp messages. For reference, the following system-generated messages are sent within WhatsApp (not in the ScaleCore UI):

| Element | Copy | Channel |
|---------|------|---------|
| Memory cleared confirmation | Sent via WhatsApp when lead types `#limpar` | WhatsApp (server-generated) |
| Insufficient credits | Agent silently stops responding (no message sent) | None (logged server-side) |

No empty states, error states, or destructive confirmations are needed for this phase since there are no new UI screens.

## Incidental UI Guidance

If any plan in this phase introduces minor UI touches (not currently anticipated), follow these rules:

1. **Agent status indicators** (e.g., "processing", "insufficient credits"): Use `Badge` component with `muted` variant. Do not introduce new status colors.
2. **Wallet balance warnings**: Use existing `destructive` color token only. Display inline within the agent card, not as a toast.
3. **Any new component**: Must use existing Radix UI primitives from `components/ui/`. Do not install new shadcn components without updating this spec.
