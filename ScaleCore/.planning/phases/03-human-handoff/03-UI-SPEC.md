## Component Inventory

### 1. Escalation Keywords Input (HAND-03)

**Location:** `components/agents/agent-form.tsx`, new section between "Tag de Ativacao" and "Instancias WhatsApp" sections.

**Pattern:** Tag-entry input (type keyword, press Enter to add, click X to remove). Follows the same visual convention as the existing `tag-input.tsx` component, but for multiple values.

**Layout:**
```
Section heading: "Palavras de Escalacao"  (text-lg, normal weight)
Helper text below heading                  (text-xs, muted-foreground)
[ Input field with placeholder ]           (full width, same as other inputs)
[ badge ] [ badge ] [ badge ]              (flex wrap, gap-2, below input)
```

**Specifications:**

| Property | Value |
|----------|-------|
| Section heading | `text-lg` (18px), weight 400 |
| Input | Standard `<Input>` component, full width |
| Input placeholder | `ex: falar com atendente` |
| Helper text | `text-xs text-muted-foreground`, displayed below input |
| Keyword badges | `<Badge variant="secondary">` with X close button |
| Badge gap | `gap-2` (8px) between badges, `flex flex-wrap` |
| Badge close icon | `X` from lucide-react, `h-3 w-3`, `ml-1` inside badge |
| Max keywords | No hard UI limit; validation at API level |
| Add behavior | Press Enter to add keyword; input clears after add |
| Remove behavior | Click X on badge to remove keyword |
| Duplicate prevention | If keyword already exists, show toast error: "Essa palavra ja foi adicionada" |
| Empty state | No badges shown; only input + helper text visible |

**Separator:** Add `<Separator />` after the escalation keywords section, before the "Instancias WhatsApp" section. Matches existing form structure.

### 2. System Note Message (HAND-02)

**Location:** `components/chat/chat-window.tsx`, extends existing message type handling. Also requires update to `components/chat/system-message.tsx` or a new sibling component.

**Pattern:** The existing `SystemMessage` component renders `message_type === "system"` as a centered gray pill. The handoff summary (`message_type === "system_note"`) needs a distinct treatment because it contains multi-line AI-generated content (3-5 sentences).

**Layout:**
```
+------------------------------------------+
| [Bot icon]  Resumo da conversa           |
|                                          |
| Summary text paragraph from LLM...       |
| Multiple lines, 3-5 sentences.           |
|                                          |
|                              14:32       |
+------------------------------------------+
```

**Specifications:**

| Property | Value |
|----------|-------|
| Container | `mx-auto max-w-[85%]`, centered in chat timeline |
| Background | `bg-muted/40` (subtle, lighter than chat bubbles) |
| Border | `border border-border` (matches card convention) |
| Border radius | `rounded-lg` (--radius: 0.75rem) |
| Padding | `px-4 py-3` (16px horizontal, 12px vertical) |
| Header row | Flex row: icon + label text |
| Icon | `Bot` from lucide-react, `h-4 w-4 text-muted-foreground` |
| Header label | `text-xs font-medium text-muted-foreground`, text: "Resumo da conversa" |
| Header gap | `gap-2` (8px) between icon and label |
| Summary text | `text-sm text-muted-foreground mt-2`, `whitespace-pre-wrap` |
| Timestamp | `text-[10px] text-muted-foreground/70 mt-1 text-right` |
| Content prefix | Strip `[Resumo IA] ` prefix from content before rendering (the prefix is for DB identification, not display) |
| Fallback display | If content starts with `[Handoff IA]`, render with `text-muted-foreground/50` (dimmer) to indicate incomplete summary |

**Integration point:** In `chat-window.tsx`, add a condition for `message_type === "system_note"` alongside the existing `message_type === "system"` check (line 1008). Both can share the same `React.Fragment` + `DateSeparator` wrapper pattern.

## Interaction Contracts

### Keyword Entry Flow

1. User types keyword text in input field
2. User presses Enter
3. IF keyword is empty: show toast "Digite uma palavra antes de adicionar"
4. IF keyword already exists (case-insensitive match): show toast "Essa palavra ja foi adicionada"
5. IF valid: keyword appears as Badge below input, input clears, focus returns to input
6. User clicks X on badge to remove keyword
7. Keywords are stored as `string[]` in form state (field name: `escalation_keywords`)

### System Note Display Flow

1. Handoff occurs (server-side: tag removed + summary generated)
2. Summary inserted into `messages` table with `message_type: "system_note"`
3. Chat window receives message via existing real-time subscription
4. Message renders as system note card (not a chat bubble)
5. No user interaction needed (read-only display)
